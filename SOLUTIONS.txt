SOLUTIONS FOR BUGS LISTED IN BUGS.txt

Date: 2026-03-24

Notes:
- This file maps directly to the numbered items in BUGS.txt.
- Some items are code fixes, some are cleanup tasks, and some are consistency/security hardening steps.

1. Registration calls a method that does not exist
Solution:
- In classes/User.php, keep one canonical method name for email existence checks.
- Either rename email_exists() to emailExists(), or add a wrapper method emailExists() that calls email_exists().
- Update process/process_register.php to use the same method name consistently everywhere.

2. Landlord registration redirects to a deleted/nonexistent page
Solution:
- Change the landlord success redirect in process/process_register.php from ../views/landlord-profile.php to ../landlord/landlord-profile.php.
- Search the repo for any remaining references to views/landlord-profile.php and replace them with the active landlord route.

3. Add-property process still prints debug output before redirects
Solution:
- Remove echo "<pre>", print_r(...), and any other debug output from process/process_add_property.php.
- Ensure no output is sent before any header() call in process files.

4. Add-property debug print can raise notices when the submit key is missing
Solution:
- Stop reading $_POST['add_property'] before checking isset($_POST['add_property']).
- Validate the request method first, then validate required POST keys.

5. Inspection process uses the wrong guard filename casing
Solution:
- Replace require_once '../UserGuard.php' with require_once '../userguard.php' in process/process_inspection.php.
- Standardize file-name casing across the project so Linux deployments do not fail.

6. Inspection validation redirects to the wrong property-details query parameter
Solution:
- Change the redirect in process/process_inspection.php from property-details.php?id=... to property-details.php?property_id=....
- Search for other redirects to property-details.php and make sure they all use property_id.

7. Inspection cancel action is done via GET and lacks ownership verification
Solution:
- Convert cancel inspection to a POST form.
- Add a server-side ownership check before update_inspection_status() runs.
- Ideally create a method like get_inspection_by_id_and_user($inspection_id, $user_id) in classes/Inspection.php and verify it before cancel/reschedule.

8. Landlord inspection query joins on the wrong property key
Solution:
- In classes/Inspection.php, change JOIN properties p ON i.property_id = p.id to JOIN properties p ON i.property_id = p.property_id.
- Test landlord inspection retrieval after the query is corrected.

9. Property-details page dereferences property data before confirming the property exists
Solution:
- In views/property-details.php, move the "if (!$details)" check so it runs immediately after get_property_by_id().
- Only fetch landlord/user details after confirming $details is valid.

10. Admin landlord-details page is completely blocked by debug output and die()
Solution:
- Remove the debug block (print_r + die) from admin/views/landlord-details.php.
- Replace the hardcoded ID lookup with the actual request parameter and add proper admin authorization.
- Decide whether the page should stay and be completed, or be removed if unused.

11. Admin tenant-fetch endpoint checks the wrong session keys
Solution:
- In admin/process/process_get-tenant.php, use the admin session contract already used elsewhere:
  - admin_id
  - admin_role
  - is_admin
- Remove checks that expect regular user session keys.
- Align role naming with the actual admin roles used during login.

12. User-details modal builds an edit URL with a field that does not exist
Solution:
- In admin/views/user-details.php, replace user.user_id with user.id in the JS template, or return user_id from the backend if that is the intended field.
- Also confirm that edit-user.php actually exists before linking to it.

13. Wishlist toggle trusts HTTP_REFERER blindly
Solution:
- In process/process_wishlist.php, add a safe fallback redirect such as ../views/properties.php if HTTP_REFERER is missing.
- Optionally validate that the referrer belongs to the current site before redirecting.

14. Public nav still uses GET logout while other areas moved toward safer POST logout
Solution:
- Convert logout in partials/nav.php to a POST form, matching the safer approach already used elsewhere.
- Keep process/process_logout.php tolerant during migration if needed, but standardize on POST in the UI.

15. Register view has duplicate DOM ids across the login and register forms
Solution:
- Give each form field a unique id, for example:
  - login_email / register_email
  - login_password / register_password
- Update label for="" attributes and any JavaScript selectors that depend on those ids.

16. Register page still contains dead placeholder links
Solution:
- Replace the forgot-password placeholder with a real route if password reset exists.
- If password reset does not exist yet, remove or visibly disable the control.
- The auth-switch links can remain JS-driven, but they should use button semantics or href values that do not look like unfinished links.

17. Admin login page still contains a dead forgot-password link
Solution:
- Same rule as the public register page:
  - connect it to a real admin reset flow, or
  - remove/disable it until that flow exists.

18. Contact page still contains dead social links
Solution:
- Replace # with real URLs.
- If the social profiles do not exist yet, remove the icons or render them in a disabled/non-clickable state.

19. Soft-delete filtering logic is fragile and may leak deleted rows if data becomes inconsistent
Solution:
- Replace filters like (deleted_at IS NULL OR status <> 'deleted') with a stricter rule, usually:
  - deleted_at IS NULL AND status <> 'deleted'
- Apply the same rule consistently anywhere deleted properties are excluded from stats, tables, and application queries.

20. Admin/user/session model is still inconsistent across parts of the codebase
Solution:
- Define a clear session contract for:
  - regular users
  - tenants/landlords
  - admins
- Update all auth-sensitive code to rely only on those agreed keys.
- A small shared helper for auth/session checks would reduce repeated mistakes.

21. There are multiple leftover debug/dead-code artifacts in production paths
Solution:
- Remove all live print_r, var_dump, die, and development echo blocks from request handlers and pages.
- Keep debugging behind comments, logs, or environment checks only.
- Audit both classes/dev.php and admin/classes/dev.php and either remove them or isolate them from production code paths.

22. The repo still contains large placeholder/demo pages with dead links
Solution:
- Review pages like views/DASHBOARD-2.php and either:
  - delete them if unused, or
  - move them out of the production path.
- Do not leave demo pages tracked in active routes unless they are fully wired and intentional.
