🔍 WHAT IS TRANSACTION HANDLING?
=================================

Transaction handling ensures data integrity when performing multiple database operations that must succeed or fail together.

**CURRENT PROBLEM:**
In your `apply()` method, you do:
1. Insert application into `applications` table
2. Update statistics in `property_stats` table via `track_application()`

If step 1 succeeds but step 2 fails, you'll have inconsistent data - an application exists but statistics don't reflect it.

**TRANSACTION SOLUTION:**
```php
public function apply($property_id, $user_id, $message){
    try{
        // Start transaction - all operations must succeed or none will
        $this->dbconn->beginTransaction();
        
        // ... validation code ...
        
        // Step 1: Insert application
        $sql = "INSERT INTO applications(property_id, user_id, `message`, status) VALUES (?, ?, ?, 'pending')";
        $stmt = $this->dbconn->prepare($sql);
        $stmt->execute([$property_id, $user_id, $message]);
        
        // Step 2: Update statistics
        $this->track_application($property_id);
        
        // Commit both operations permanently
        $this->dbconn->commit();
        return "success:application_submitted";
        
    }catch(Exception $e){
        // If ANY operation fails, undo everything
        $this->dbconn->rollBack();
        error_log("Application transaction failed: " . $e->getMessage());
        return "error:system_failure";
    }
}
```

**REAL-WORLD EXAMPLE:**
- Banking transfer: Debit Account A AND credit Account B must both succeed
- Your application: Insert application AND update statistics must both succeed

**BENEFITS:**
- ✅ Data consistency - no partial updates
- ✅ Error recovery - automatic rollback on failure  
- ✅ Concurrent access safety - prevents race conditions

**WHEN TO USE:**
- Multiple INSERT/UPDATE/DELETE operations that are related
- When data must stay consistent across tables
- Operations that shouldn't be partially completed
